# AlexManus 项目改造说明（评委版）

## 1. 这个项目到底在做什么

一句话概括：  
**AlexManus 是在 `https://github.com/kortix-ai/suna` 基础上，面向中文业务场景和本地可控部署需求做的“可落地智能体工作平台”。**

它不是只会聊天的 AI 页面，而是一个能把“用户目标 -> 多步骤执行 -> 工具调用 -> 结果回传”完整跑通的系统。  
用户可以把它当作“会研究、会查资料、会调工具、会持续执行任务”的数字执行助手。

---

## 2. 为什么要在 Suna 基础上改造

Suna 的 Agent 架构很优秀，但在真实项目落地时，团队通常会遇到三类问题：

1. 依赖国外服务或固定生态，私有化和本地化改造成本高。
2. 从“能演示”到“能稳定跑业务”之间，缺少对会话状态、流式输出、停止控制、权限隔离的工程增强。
3. 对中文团队的模型、检索、沙箱、运维习惯支持不够直接。

AlexManus 的改造目标就是：  
**保留 Suna 的 Agent 核心能力，同时把系统改造成更贴近国内团队、可持续运行、可解释可维护的产品形态。**

---

## 3. 我们基于 Suna 做了哪些关键改造

### 3.1 从“演示型 Agent”到“会话型任务系统”

我们把业务过程拆成了更清晰的三层：

1. `Thread`：会话上下文（用户持续对话容器）。
2. `Agent Run`：一次具体执行实例（可运行、可停止、可追踪状态）。
3. `Stream`：实时输出通道（前端能持续看到执行过程）。

这意味着系统不再是“一问一答”的松散交互，而是可管理的任务生命周期。

### 3.2 重新设计了前后端协作链路

在用户体验上，支持两条高频路径：

1. 新建任务：从首页输入目标，系统自动创建项目/会话并启动执行。
2. 继续任务：进入历史会话，补充新指令并继续运行。

这种设计更符合真实工作流，例如“上午先做调研，下午继续补充分析”，而不是每次都从零开始。

### 3.3 强化了流式执行与可中断能力

我们引入了“Redis 列表 + 发布订阅 + SSE”的组合机制：  
后台持续写入执行结果，前端实时接收；用户点击停止时，系统可把停止信号精准送到正在运行的实例。

这类能力对评委最直观的价值是：  
**不是等 AI 一次性吐答案，而是像看一个助理实时工作，且可以随时叫停、换方向。**

### 3.4 认证与权限模型改造成“更可控”

相对原始生态依赖，我们采用了本地化 JWT 认证与会话管理方案，并在线程/运行级别做访问校验。  
这让系统在企业或教学环境中更容易做账号隔离和权限边界管理。

### 3.5 基础设施从“依赖型”转向“可部署型”

我们将核心运行栈明确为：

1. FastAPI（API 网关与业务编排）
2. PostgreSQL（事实数据与状态）
3. Redis（异步协同与流式通知）
4. Worker（后台执行）

这样可以更直接地落到本地机房、实验室服务器或私有云环境，而不被单一外部平台绑定。

### 3.6 本地化生态适配（模型与工具）

我们加强了对中文团队常用能力的兼容：

1. 多模型来源切换（如 DeepSeek 相关 provider 路由）。
2. 网络检索、网页抓取、沙箱执行等工具链整合。
3. 面向中文用户的交互文案与产品流程。

结果是：用户可以用熟悉的中文工作方式驱动一个“会调工具的 Agent 系统”。

## 4. 和 ChatGPT 深度搜索的关系：不是“替代”，而是“能力补位”

评委常见问题是：  
“现在 ChatGPT 深度搜索已经很强了，为什么还需要你这个项目？”

我们的答案不是“谁更聪明”，而是“谁更适合你的业务边界”。

| 维度 | 通用深度搜索产品（以 ChatGPT 类产品为代表） | AlexManus |
|---|---|---|
| 目标定位 | 面向广泛个人用户，强调通用问答与检索体验 | 面向团队/组织，强调可部署、可管控、可持续执行 |
| 数据边界 | 通常是平台托管模式 | 可本地/私有环境部署，数据边界可自定义 |
| 执行可控性 | 用户更关注“答案结果” | 支持“会话-运行-流式”全链路控制，可中断、可追踪 |
| 系统集成 | 偏产品内闭环 | 更适合对接企业内部流程与工具链 |
| 模型策略 | 由平台统一提供 | 可按成本、性能、合规要求灵活切换模型与 provider |

一句话：  
**ChatGPT 深度搜索解决“快速得到高质量答案”，AlexManus 解决“让组织把 AI 能力纳入可控生产流程”。**

## 5. 这些改造在真实场景里怎么体现

### 场景 A：行业研究与竞品追踪

产品经理输入“调研某赛道最近 30 天动态并形成结论”。  
系统会进入连续执行：检索、抓取、提炼、回传。  
如果方向偏了，用户可中途停止并补充新要求，不必推倒重来。

### 场景 B：运营同学的内容与情报处理

运营人员把多来源素材交给系统，让它先做结构化整理，再输出可直接使用的摘要或执行建议。  
相比单次问答，AlexManus 更像“可持续协作的执行助手”。

### 场景 C：教学与训练环境

在课程或内部培训中，学员能看到完整链路：  
从任务创建、实时执行、工具轨迹到结果回放，便于理解智能体系统如何真正落地。

## 6. 评委可能会问的“专业刁钻问题”与回答

### Q1：ChatGPT 深度搜索效果可能更好，为什么还需要 AlexManus？

**回答要点：**  
我们不是和 ChatGPT 做“单次答案质量”的一维竞争，而是解决“组织级落地”的问题。  
很多团队的真实痛点是：数据不能外流、流程要留痕、执行要可中断、模型成本要可控、系统要能接内网工具。  
AlexManus 把这些能力组合成一套可运行体系，这正是通用产品难同时覆盖的部分。

### Q2：你说市面产品做不到，具体是哪些“做不到”？

更准确地说，不是“绝对做不到”，而是**很难同时做到以下组合**：

1. 私有部署与数据边界可控。
2. 可持续会话执行（不是一次性问答）。
3. 运行级状态管理（启动/流式/停止/回放）。
4. 模型与工具链可替换、可审计。

单点能力市场上很多产品都有，但“组合能力 + 业务可落地”才是我们的核心改造价值。

### Q3：如果用户只是偶尔查资料，用 ChatGPT 不就够了吗？

是的，轻量场景下通用产品就够。  
AlexManus 的价值在“高频、持续、可管理”的任务场景：  
例如持续竞品追踪、周期性运营情报整理、内部知识执行流程等。  
当需求从“偶尔问一问”升级到“稳定产出 + 团队协作 + 审计留痕”，平台价值会明显放大。

### Q4：这个项目的技术壁垒到底在哪里？

壁垒不在某个单点算法，而在工程体系：

1. 会话与运行实例分层（Thread / Agent Run）。
2. Worker + Redis + SSE 的实时执行链路。
3. 停止控制、状态收敛与错误恢复机制。
4. 认证权限与部署形态的可控性。

这类能力决定了它是“可长期运行的系统”，而不是“漂亮的 Demo 页面”。

### Q5：如何证明它已经具备业务可用性，而不是概念验证？

可用性验证建议关注三类指标：

1. 任务完成链路：创建会话 -> 执行 -> 中断 -> 继续执行是否稳定闭环。
2. 运维可观测：是否能定位到 run 级别的状态和异常。
3. 成本与策略：是否能根据场景切换模型与 provider。

这三点都能在当前项目架构中找到对应实现，不依赖单次人工演示。

## 7. 评委最关心的“改造价值”总结

如果只看技术名词，这像是一次架构升级；  
如果站在业务视角，它本质上完成了三件事：

1. **可用性提升**：从“能对话”变成“能持续完成任务”。
2. **可控性提升**：状态可追踪、过程可中断、权限可管理。
3. **可落地性提升**：本地化部署更友好，适配中文团队的模型与工具生态。

因此，AlexManus 不是简单“复刻 Suna”，而是基于 Suna 的能力底座，做了面向真实使用场景的产品化和工程化改造。
